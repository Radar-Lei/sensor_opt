<decisions>
## Implementation Decisions

### Runtime Blocker Disposition
- **D-01:** Phase 8 real Stage12 execution is blocked by runtime feasibility, not by a need to keep waiting.
- **D-02:** Do not advance Phase 9, Phase 10, paper claims, or external evidence elevation until Phase 8.5 proves full-seed feasibility for PeMS7_1026 and Seattle.
- **D-03:** Downscaled profiling and diagnostic runs may guide implementation but must remain explicitly non-evidence.

### Required Runtime Fixes
- **D-04:** Add a fast validation path so RCSS candidate validation computes only `args.selection_method` MAE and does not run full `evaluate_layout` or certificate computation.
- **D-05:** Add or preserve a `solve_quadratic` fast path for scalar/vector-equivalent observation weights so no-missing validation avoids per-timestep dense solves.
- **D-06:** Use Cholesky-based solves where valid for SPD linear systems instead of repeated generic dense solves.
- **D-07:** Reduce repeated posterior/scenario trace full inversions with caching or Woodbury/Sherman-Morrison-style low-rank updates where numerically appropriate.
- **D-08:** Progress logging/checkpointing must make long runs diagnosable by budget, candidate batch, and validation-swap iteration before a full seed completes.

### Optional Algorithmic Change Fence
- **D-09:** Validation-swap top-K proxy filtering is optional and should be planned only after runtime-only safe fixes. If introduced, validate it first on PeMS7_228 against the existing Stage12 behavior because it can affect candidate search semantics.
- **D-10:** Any optimization that changes selected layouts or scores beyond documented numerical tolerance must be labeled as a method/search change, not a pure runtime fix.

### Acceptance Gate
- **D-11:** Phase 8.5 passes only when one Stage12-compatible full seed for PeMS7_1026 and one for Seattle complete and aggregate successfully.
- **D-12:** Those one-seed outputs prove feasibility for rerun planning but do not satisfy EVID-03/EVID-04 ten-split evidence.
- **D-13:** The external evidence gate must remain fail-closed until complete ten-split aggregate evidence exists.

### Claude's Discretion
- Choose exact helper names, cache class/function boundaries, and smoke-test commands following existing Python script style.
- Prefer targeted edits in `TRC-23-02333/transparent_estimator_eval.py` and existing launchers over creating a duplicate pipeline.

</decisions>

<specifics>
## Specific Ideas

- Prioritize Patch A first: `validation_mae()` should compute only historical, neighbor, GSP, or GLS/MAP selected-method MAE directly.
- In no-missing validation, pass scalar observation weight into `solve_quadratic` instead of a dense T×N weight matrix.
- `solve_quadratic` should recognize T×N weights whose sensor columns are constant over time and collapse them to vector/scalar-equivalent weights.
- Use `scipy.linalg.cho_factor` / `cho_solve` for SPD systems where existing matrices support it.
- Cache posterior trace/logdet components once per base/scenario matrix and evaluate layout traces through small selected-sensor systems.
- Progress outputs should be durable enough that a killed long run leaves evidence of the last budget/candidate/swap stage reached.
- Resource tuning can test `MAX_JOBS=1` and `THREADS_PER_JOB=4` for large dense solves, but resource tuning alone is not a substitute for code-level fixes.

</specifics>
