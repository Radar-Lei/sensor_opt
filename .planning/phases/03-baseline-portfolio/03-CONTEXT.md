<domain>
## Phase Boundary

This phase builds a reviewer-resistant TRACE-SL baseline portfolio so the strong Transportation Science claim can be tested against OR/TSLP-style, A/D-optimal design, graph sampling/GSP, QR/SVD/POD sparse placement, learning-style reconstruction, and multistart validation-refinement alternatives.

The phase should implement or document baseline coverage in the existing TRACE-SL experiment pipeline, make independent baseline rows visible in generated artifacts, and define defensible run commands or deferrals. It should not regenerate the full core evidence package across all final splits, run robustness stress tests, curate Seattle as core evidence, or write the final manuscript; those belong to later phases.

</domain>

<decisions>
## Implementation Decisions

### Baseline Coverage and Reviewer Risk
- **D-01:** Treat the baseline portfolio as claim defense, not feature expansion: every baseline must map to BASE-01..BASE-06 and to a reviewer objection it answers.
- **D-02:** Prefer actual evaluator support for feasible baselines over prose-only justification; prose deferral is acceptable only for learning-style checks or incompatible objectives when documented with a clear reason.
- **D-03:** Keep final claims reconstruction-centered: TSLP/observability-style baselines may optimize a different objective, but they must still be evaluated through the same held-out hidden-node reconstruction metrics for comparability.
- **D-04:** Do not weaken the strong TRACE-SL claim; add baselines and caveat handling so later evidence can support or bound it.

### Implementation Scope
- **D-05:** Reuse `TRC-23-02333/transparent_estimator_eval.py` as the source of truth and add baseline helpers/CLI flags there rather than creating a copied evaluator.
- **D-06:** Preserve existing artifact schemas where possible; new baselines should appear as additional `layout_type` rows evaluated by existing reconstruction methods, especially `gls_map`.
- **D-07:** Add run-script wiring only where it establishes a reusable paper-facing baseline stage; avoid broad refactors or modularization in this phase.
- **D-08:** Validate with small smoke runs and artifact-schema checks before any full experiment recommendation.

### Baseline-Specific Decisions
- **D-09:** Implement or expose A-optimal and D-optimal design as independent reported baselines, separate from RCSS candidate sources.
- **D-10:** Cover graph sampling/GSP by either a compatible explicit graph sampling layout baseline or a documented tie to existing GSP reconstruction evidence; prefer implementation if it can reuse Laplacian/graph-distance primitives safely.
- **D-11:** Implement QR/SVD/POD sparse sensor placement where compatible with the traffic matrix representation, using training traffic modes/data and evaluating selected sensors through the standard hidden-node reconstruction pipeline.
- **D-12:** For observability/TSLP-style coverage, implement a transparent proxy such as coverage/degree/trace-style observability where exact deterministic full-observability TSLP does not match partial reconstruction; document the objective mismatch.
- **D-13:** Learning-based reconstruction should be a simple sanity check or documented deferral, not a new black-box method centerpiece. If implemented, keep it lightweight and use the same selected-sensor/test split discipline.
- **D-14:** Multistart validation refinement must become a named comparator or predeclared portfolio member so the 10% PeMS7_228 issue cannot be read as post-hoc cherry-picking.

### Evidence and Reporting Contract
- **D-15:** Phase 3 should produce a baseline inventory/coverage artifact that says which BASE requirement is implemented, already represented, deferred, or requires Phase 4 full evidence.
- **D-16:** Baseline names must be stable and paper-readable so Phase 4 aggregation and Phase 7 manuscript writing can cite them without renaming.
- **D-17:** Validation MAE remains selection evidence only; baseline claims must ultimately rely on held-out test metrics generated by the evaluator and later audited in Phase 4.
- **D-18:** Raw datasets stay local and ignored; do not read or commit dataset contents while planning baseline support.

### Claude's Discretion
The planner may choose the exact order of baseline implementation and whether some baselines are grouped under one CLI flag, as long as every BASE-01..BASE-06 item has an explicit implementation, evidence hook, or defensible deferral. The planner should bias toward minimal, compatible additions to the existing evaluator over architectural cleanup.

### Established Patterns
- The repository is a script-oriented research pipeline; add CLI flags and helper functions in the monolithic evaluator rather than creating parallel scripts.
- Layouts are evaluated by generating sensor sets, assigning stable `layout_type` names, and passing them through `evaluate_layout()` for reconstruction methods including `gls_map`.
- Validation-based selection and held-out test evaluation are separated; preserve this distinction in baseline naming and summaries.
- Result artifacts are CSV/JSON/Markdown under `TRC-23-02333/trace_sl_results/`; raw datasets under `TRC-23-02333/dataset/` remain ignored and should not be committed.
- There is no formal test framework; validation relies on deterministic smoke runs and checking generated artifacts.

### Integration Points
- Primary implementation target: `TRC-23-02333/transparent_estimator_eval.py`, near existing layout/search functions and the budget loop.
- Aggregation target: `TRC-23-02333/summarize_trace_sl_rcss.py` only if new baseline rows need additional focused summaries beyond the existing layout-type grouping.
- Launcher target: `scripts/` if a repeatable Stage 12/Phase 3 baseline portfolio run script is useful.
- Planning/evidence target: `.planning/phases/03-baseline-portfolio/` for baseline inventory, plan, summaries, verification, and any UAT artifact.

- Keep learning-based reconstruction lightweight or deferred so it does not shift the contribution toward black-box imputation.
